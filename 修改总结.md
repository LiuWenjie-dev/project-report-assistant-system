## Jira表单登录修复（关键更新）

### 问题
用户反馈：即使输入正确的用户名和密码也无法登录成功。原因是之前的验证使用API token方式，但Jira实际使用表单登录。

### 解决方案
1. **分析Jira登录页面结构**：
   - 用户名字段：`<input name="os_username" type="text">`
   - 密码字段：`<input name="os_password" type="password">`

2. **实现表单登录验证**：
   - 修改 `jira_client.py` 中的 `test_connection_with_credentials` 方法
   - 使用 `requests.Session()` 维护会话状态
   - 模拟浏览器登录流程：
     a. 访问登录页面获取CSRF token
     b. 提交表单数据（os_username, os_password）
     c. 处理重定向和会话cookie
     d. 验证登录是否成功

3. **关键代码改进**：
   ```python
   def test_connection_with_credentials(self, base_url, username, password):
       # 创建会话
       session = requests.Session()

       # 访问登录页面
       login_url = urljoin(base_url, "/login.jsp")
       response = session.get(login_url)

       # 提交登录表单
       login_data = {
           "os_username": username,
           "os_password": password,
           "os_destination": "",
           "user_role": "",
           "atl_token": "",
           "login": "登录"
       }

       # 提交表单
       response = session.post(login_url, data=login_data)

       # 验证登录是否成功
       return "登录成功" in response.text or "dashboard" in response.url
   ```

4. **优势**：
   - 使用真实的表单登录，与浏览器行为一致
   - 处理会话cookie和重定向
   - 支持企业级Jira认证流程
   - 提高登录验证的准确性和可靠性

### 最终验证结果
- ✓ 现在使用表单登录（os_username/os_password）而不是API token
- ✓ 正确的用户名和密码可以成功登录
- ✓ 错误的凭据会被正确拒绝
- ✓ 登录验证与Jira网页版行为一致

## Jira登录验证修复（最终版本）

### 发现的问题

2. **CSRF token处理**：没有正确提取和使用 `atl_token`
3. **登录失败判断**：无法准确检测登录失败
4. **重定向处理**：没有正确处理登录后的重定向

### 修复方案
1. **改进表单字段提取**：
   - 从登录页面动态提取所有表单字段
   - 正确处理CSRF token（`atl_token`）
   - 自动包含所有隐藏字段

2. **增强验证逻辑**：
   - 检查HTTP重定向状态码（301/302/303）
   - 分析重定向目标URL
   - 检查响应内容中的错误关键词
   - 验证是否仍然在登录页面

3. **多维度验证**：
   ```python
   # 1. 检查重定向
   if response.status_code in [301, 302, 303]:
       location = response.headers.get('location', '')
       if 'dashboard' in location.lower():
           return True  # 成功重定向到dashboard
       if 'login' in location.lower():
           return False  # 重定向回登录页面

   # 2. 检查错误信息
   error_keywords = ['sorry', 'invalid', 'incorrect', 'error', 'failed']
   for keyword in error_keywords:
       if keyword in response_text:
           return False

   # 3. 检查是否仍然在登录页面
   if 'login' in response.url and contains_login_form(response.text):
       return False

   # 4. 检查成功迹象
   success_keywords = ['welcome', 'dashboard', 'log out', 'sign out']
   for keyword in success_keywords:
       if keyword in response_text:
           return True

   # 5. 尝试访问需要认证的页面
   profile_response = session.get("/secure/ViewProfile.jspa")
   if profile_response.status_code == 200:
       if username in profile_response.text:
           return True

   # 6. 默认假设失败（更安全）
   return False
   ```

4. **错误处理和日志**：

   - 改进异常处理
   - 添加超时控制

## 近期关键修复补充

### 1. Jira筛选JQL生成修复

#### 问题
组合筛选时，JQL生成不符合当前Jira实例规则，导致查询失败：
- Project ID 一度被自动发现为 `cf[12300]`，但实际可用JQL字段是 `"Project ID"`
- Project ID 值被错误加引号，实际可用写法是 `"Project ID" = BM20CA-S905X5M`
- 优先级曾被截断为 `P0/P1`，但当前Jira优先级真实值是 `Highest/High/Medium/Low/Lowest`
- ID + 优先级/标签组合筛选时容易生成无效JQL

#### 解决方案
1. **固定Project ID字段写法**：
   - 修改 `data_sources/jira_client.py`
   - `get_project_id_jql_field()` 固定返回 `"Project ID"`
   - 不再使用自动发现出的 `cf[12300]` 作为查询字段

2. **统一JQL值格式化**：
   - 修改 `gui/dashboard_window.py`
   - 新增 `_format_jql_value()`
   - 简单ID值保持不加引号，例如：`BM20CA-S905X5M`
   - 含空格或特殊字符的值自动加引号，例如：`"P0 2"`

3. **统一条件构建**：
   - 新增 `_build_jql_condition()`
   - Project ID、priority、status、labels 共用同一套JQL条件构建逻辑
   - 多值筛选会生成 `(a OR b)` 结构

4. **优先级映射**：
   - 新增 `_normalize_priority_values()`
   - 支持将内部P级别习惯名映射为Jira标准优先级：
     - `P0 -> Highest`
     - `P1 -> High`
     - `P2 -> Medium`
     - `P3 -> Low`
     - `P4 -> Lowest`
   - 直接输入 `High/Highest/Medium/Low/Lowest` 时保持原样

#### 当前验证结果
```text
"Project ID" = BM20CA-S905X5M
priority = Highest
priority = High
"Project ID" = BM20CA-S905X5M AND priority = Highest
```
